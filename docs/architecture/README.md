# NL Table: карта системы и границы модулей

Статус: архитектурная фиксация перед рефакторингом.

Дата: 2026-06-11.

Эти документы описывают текущую систему и целевые границы модулей. Они не
предписывают переход на микросервисы: NL Table остается модульным монолитом на
FastAPI, PostgreSQL, Redis и Celery.

## Документы

- `system-map.html` - визуальная карта приложения, инфраструктуры и связей.
- `data-flows.md` - основные потоки данных от WB API до интерфейса.
- `module-boundaries.md` - владельцы маршрутов и таблиц, допустимые зависимости,
  технический долг и порядок выноса из `api/v1/nl.py`.
- `visual-baseline.md` - перечень закрытых эталонных снимков, их контрольные
  суммы и правила повторной визуальной проверки.
- `../security/endpoint-audit-2026-06-11.md` - аудит авторизации,
  межтенантной изоляции и системных настроек endpoints.

## Главный принцип

Каждая бизнес-область получает одного владельца логики и записи данных:

```text
Frontend -> API router -> application service -> repository/model -> PostgreSQL
                              |
                              +-> WB API client

Celery task -> application service
```

Роутеры отвечают за HTTP-контракт, сервисы - за сценарии и бизнес-правила,
репозитории - за доступ к данным, задачи Celery - только за запуск и расписание.

## Ограничения пункта 2

- Рабочий код и поведение API не меняются.
- Схема БД и данные не меняются.
- Контейнеры и расписание Celery не меняются.
- Найденные дубли и нарушения границ только фиксируются.
