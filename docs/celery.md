# Celery

## Точка входа

Worker и Beat используют одно приложение:

```bash
celery -A tasks.celery_app:celery_app worker --loglevel=info
celery -A tasks.celery_app:celery_app beat --loglevel=info
```

Конфигурация приложения находится в `tasks/celery_app.py`, расписание -
в `tasks/celery_schedule.py`. Часовой пояс приложения: `Europe/Moscow`;
значения `crontab` задаются сразу в московском времени.

## Расписание

- 03:05 - товары;
- 03:15 - склады;
- 03:20 - остатки FBS;
- 03:23 - остатки FBO;
- 08:00 - продажи;
- 08:05 - заказы;
- 08:30 - коробочные тарифы;
- 09:30, 15:30, 22:00 - разбор raw-данных;
- 14:00 - общий срез остатков;
- 14:03 - остатки FBO;
- 14:05 - продажи;
- 20:00 - тарифы;
- 20:05 - рекламные кампании;
- 21:00 - рекламная статистика;
- 21:30 - воронка продаж;
- 23:00 - тарифный снимок;
- каждый час в `:30` - цены;
- каждые два часа - акции;
- каждый час в `:45` - проверка свежести raw-данных.

## Надежность

- WB 429 обрабатывается существующим retry/backoff с `Retry-After` и jitter.
- Worker подтверждает задачу после выполнения и берет по одной задаче.
- Мягкий лимит задачи - 50 минут, жесткий - 60 минут.
- Все запуски записываются в `celery_task_runs`.
- Вложенные результаты со статусом `error`, `failed` или `stale` получают
  статус `warning`, даже если Python-функция завершилась без исключения.
- Ошибки всегда попадают в журнал worker. Если задан
  `CELERY_ALERT_WEBHOOK_URL`, туда отправляется JSON-уведомление.

## Деплой

Миграция должна быть применена до запуска новой версии worker:

```bash
alembic upgrade head
docker compose up -d --build app celery-worker celery-beat
docker compose logs --tail=100 celery-worker celery-beat
```

На одном окружении должен работать только один экземпляр Beat. При переносе
сервера сначала остановить старый Beat, затем запускать новый.
